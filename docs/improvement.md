# 개선 방향

## 개요

이 문서는 AI Startup Investment Evaluator 프로젝트의 기술적 고도화 아이디어와 확장 전략을 제시합니다.

## 기술적 고도화 아이디어

### 1. 비동기 병렬 처리

**현재 상태**:
- 에이전트들이 순차적으로 실행됨
- 시장성 평가와 경쟁사 분석이 독립적임에도 순차 실행

**개선 방향**:
- 독립적인 에이전트(시장성 평가, 경쟁사 분석)를 병렬 실행
- 전체 실행 시간 단축
- 리소스 활용 효율성 향상

**구현 방법**:
- LangGraph의 병렬 실행 기능 활용
- asyncio를 활용한 비동기 처리
- 각 에이전트의 독립성 보장

**예상 효과**:
- 전체 실행 시간: 30-40% 단축 예상
- 리소스 활용 효율성: CPU/GPU 활용률 향상

**도전 과제**:
- State 동시 접근 문제 해결
- 에러 핸들링 및 롤백 전략 수립

---

### 2. 평가 기준 외부화

**현재 상태**:
- 7가지 평가 기준이 하드코딩되어 있음
- 도메인별 커스터마이징 어려움
- 투자 단계별 다른 기준 적용 불가

**개선 방향**:
- YAML/JSON 설정 파일로 평가 기준 및 가중치 관리
- 도메인별 평가 기준 정의 가능
- 투자 단계별 다른 기준 적용 가능

**구현 방법**:
- 설정 파일 구조 설계
- 동적 평가 기준 로딩
- 가중치 조정 기능 추가

**설정 파일 예시**:
```yaml
evaluation_criteria:
  seed_stage:
    - name: "창업자"
      weight: 1.5
      description: "창업자의 경력과 역량"
    - name: "시장성"
      weight: 2.0
      description: "시장 크기와 성장 가능성"
  # ...
  
  series_a:
    - name: "실적"
      weight: 2.0
      description: "매출 및 성장 실적"
    # ...
```

**예상 효과**:
- 유연성: 도메인별, 단계별 커스터마이징 가능
- 유지보수성: 코드 수정 없이 평가 기준 변경 가능
- 확장성: 새로운 평가 기준 추가 용이

**도전 과제**:
- 설정 파일 검증 로직 필요
- 기존 평가 결과와의 호환성 고려

---

### 3. 캐싱 전략

**현재 상태**:
- 동일 기업 재평가 시 모든 과정을 다시 수행
- VectorDB에 저장되어 있지만 LLM 호출은 매번 수행
- 비용 및 시간 낭비

**개선 방향**:
- 동일 기업 재평가 시 VectorDB 캐시 활용
- LLM 호출 최소화
- 평가 결과 캐싱

**구현 방법**:
- 평가 결과 캐시 테이블 설계
- 캐시 무효화 전략 수립
- TTL(Time To Live) 설정

**캐시 전략**:
- **기업 정보 캐시**: VectorDB에 저장된 기업 정보 재사용
- **산업 동향 캐시**: 일정 기간 내 동일 산업 동향 재사용
- **평가 결과 캐시**: 동일 기업 재평가 시 캐시된 결과 활용

**예상 효과**:
- 비용 절감: LLM 호출 횟수 감소로 비용 절감
- 실행 시간: 캐시 히트 시 실행 시간 대폭 단축
- 사용자 경험: 빠른 응답 시간으로 사용자 만족도 향상

**도전 과제**:
- 캐시 무효화 시점 결정
- 캐시 일관성 유지
- 메모리 관리

---

### 4. 도메인 확장

**현재 상태**:
- 모빌리티 도메인에 특화되어 있음
- 다른 도메인으로 확장 시 코드 수정 필요

**개선 방향**:
- 모빌리티 외 헬스케어, 핀테크 등으로 확장 가능한 구조로 리팩토링
- 도메인별 플러그인 구조 설계
- 산업별 검색 쿼리 자동 생성 개선

**구현 방법**:
- 도메인별 설정 파일 분리
- 도메인별 에이전트 플러그인 구조
- 공통 인터페이스 정의

**도메인 확장 구조**:
```
domains/
  ├── mobility/
  │   ├── config.yaml
  │   └── agents/
  ├── healthcare/
  │   ├── config.yaml
  │   └── agents/
  └── fintech/
      ├── config.yaml
      └── agents/
```

**예상 효과**:
- 확장성: 새로운 도메인 추가 용이
- 재사용성: 공통 로직 재사용 가능
- 유지보수성: 도메인별 독립적 관리 가능

**도전 과제**:
- 공통 인터페이스 설계
- 도메인별 특성 반영
- 기존 코드와의 호환성

## 우선순위

### 단기 (1-2개월)
1. **비동기 병렬 처리**: 실행 시간 단축으로 즉시 체감 가능
2. **캐싱 전략**: 비용 절감 효과가 큼

### 중기 (3-6개월)
3. **평가 기준 외부화**: 유연성 향상으로 확장성 확보
4. **실시간 모니터링**: 시스템 안정성 및 최적화

### 장기 (6개월 이상)
5. **도메인 확장**: 새로운 시장 진출 가능
6. **사용자 인터페이스**: 사용자 경험 향상
7. **고급 분석 기능**: 차별화 포인트

## 기술 부채 해결

### 1. 크롤링 의존성
**문제**: NextUnicorn 플랫폼 구조 변경 시 크롤링 로직 수정 필요

**해결 방안**:
- API 연동으로 전환 (가능한 경우)
- 크롤링 로직 모듈화 및 추상화
- 자동 테스트로 회귀 방지

### 2. LLM 비용
**문제**: 다수의 에이전트가 순차 실행되며 LLM 호출이 빈번함

**해결 방안**:
- 캐싱 전략으로 호출 횟수 감소
- 비동기 병렬 처리로 실행 시간 단축
- 더 저렴한 모델 활용 검토

### 3. 평가 기준 고정
**문제**: 7가지 평가 기준이 하드코딩되어 있음

**해결 방안**:
- 평가 기준 외부화로 유연성 확보
- 설정 파일 기반 동적 로딩

## 결론

이 프로젝트는 현재 모빌리티 도메인에 특화되어 있지만, 제시한 개선 방향을 통해 더욱 확장 가능하고 효율적인 시스템으로 발전할 수 있습니다. 특히 비동기 병렬 처리와 캐싱 전략은 즉시 체감 가능한 효과를 제공할 것으로 예상됩니다.
